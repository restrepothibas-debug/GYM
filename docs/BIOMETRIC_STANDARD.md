# Biometric Reader Standard

This project supports fingerprint readers through a vendor-agnostic adapter
contract. React components must never call vendor SDKs directly.

## Supported Provider Slots

- `mock`: local simulator for UI and QA.
- `secugen`: SecuGen WebAPI for Hamster readers.
- `digitalpersona`: HID DigitalPersona Web Client.
- `zkteco`: ZKTeco SDK through a desktop bridge.
- `suprema`: Suprema BioMini SDK through a desktop bridge.

## Adapter Contract

Every provider adapter must implement:

```js
detectDevice()
captureSample({ memberId, member })
verify({ memberId, enrollments })
identify({ enrollments, members })
```

The adapter registry lives in `src/lib/biometrics/biometricRegistry.js`.
Components use `GymContext`; they do not import SDK adapters directly.

## Security Rules

- Do not store fingerprint images.
- Do not store raw biometric templates.
- Store only encrypted template payloads generated by a trusted adapter or backend.
- Always keep `tenant_id` on biometric records.
- Require explicit member consent before enrollment.
- Revocation must clear the encrypted template payload.
- Manual/document/QR entry must remain available.

## Database

The migration `add_member_biometrics_standard` creates `public.member_biometrics`
with RLS. It stores metadata plus `template_encrypted`, which is nullable after
revocation.

Do not expose broad authenticated access. Policies must always filter by tenant
membership through `app_private.has_tenant_access`. Column grants must exclude
`template_encrypted` from `select` access; it is write-only for enrollment and
revocation from the authenticated Data API role.

The database constraint `member_biometrics_revocation_state` must reject any
`status = 'revoked'` row that keeps `template_encrypted`. UI cleanup is not
enough; scripts, adapters and API clients must be blocked by the database too.

## Hardware Rollout

1. Install vendor driver/local client on the workstation.
2. Implement the matching adapter file under `src/lib/biometrics/adapters/`.
3. Keep SDK-specific code out of React components.
4. Verify enrollment, 1:1 verification and 1:N identification with Playwright and
   a real workstation test.
5. Apply the Supabase migration before enabling remote biometric enrollment.
